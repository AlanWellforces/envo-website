# Projects Collection — Design Spec

**Date:** 2026-05-29
**Status:** Draft, pending implementation
**Pattern reference:** [2026-05-22-blog-collection-design.md](./2026-05-22-blog-collection-design.md)

## Context

`/projects` exists today as a hardcoded list page (`src/data/projects.ts` ships 4 placeholder entries: Retail Signage / Hotel Facade / Storefront / Canopy Signage). Each entry has `href: '#'` — no detail pages. The current content reads as industry categories with stock photos, not real case studies.

This spec turns Projects into a CMS-driven case-study feature mirroring the Blog implementation: Payload Collection → drafts/versions → list page → detail page → industry/tag filter routes → ISR revalidation.

## Goals

- Replace placeholder category cards with real installation case studies authored in Payload.
- Mirror Blog's authoring + delivery pattern so Wei has zero new admin learning curve.
- Cleanly handle the **0-projects-in-DB** state so we can ship before content is ready.
- Stay inside the three-source rule: Akeneo owns product data, Payload owns this editorial content, Git owns code.

## Non-Goals

- No category landing pages as a separate route layer (e.g. `/projects/retail-signage/`). Industry surfaces via filter chips + a single `/projects/industry/[industry]` route.
- No multi-project comparison / collection feature.
- No automated import from prior installation tracking systems — Wei authors directly in Payload.
- No reading-time hook (case studies aren't long-read articles).

## Schema — `Projects` Collection

Mirrors `Posts.ts` 4-tab structure, with a new "Project Details" tab for structured case-study metadata.

### Tab 1 — Content

| Field | Type | Required | Notes |
|---|---|---|---|
| `title` | text | ✓ | Real project name, e.g. *"LAX Terminal 7 Retail Signage"* |
| `slug` | text (auto, unique, readOnly) | ✓ | Generated by `beforeChange` hook from title. |
| `excerpt` | textarea (200 max) | ✓ | Card summary + meta-description fallback. |
| `cover` | upload → media | ✓ | List card + detail page hero. |
| `body` | richText | ✓ | Long-form case-study story. |

### Tab 2 — Project Details (new vs Blog)

| Field | Type | Required | Notes |
|---|---|---|---|
| `client` | text | ✗ | May be blank when client is under NDA. |
| `location` | text | ✗ | e.g. *"Los Angeles, USA"*. Free-form for now. |
| `completedYear` | number | ✗ | e.g. `2025`. |
| `gallery` | array of `{ image: upload→media, caption: text }` | ✗ | Multiple install photos with optional caption. Caption doubles as alt text + SEO copy. |
| `productsUsed` | array of `{ sku: text }` | ✗ | ENVO SKUs used in the install. Stored as strings — three-source rule keeps Akeneo data separate from Payload. Front-end resolves each SKU via `getProduct()` at render time. |
| `testimonial` | textarea | ✗ | Optional client quote. |

### Tab 3 — Taxonomy

| Field | Type | Required | Notes |
|---|---|---|---|
| `industry` | `select` with `hasMany: true` | ✓ (≥1) | Multi-select. Options: `retail`, `hotel`, `storefront`, `architectural`, `canopy`, `other`. A project can belong to multiple industries. |
| `tags` | array of text | ✗ | Free-form tags. Powers `/projects/tag/[tag]` routes. |

### Tab 4 — Publishing

| Field | Type | Required | Notes |
|---|---|---|---|
| `publishedAt` | date | ✓ | Future dates work as scheduled publishing. |
| `featured` | checkbox (default false) | ✗ | Marks a project for the "Featured" strip on `/projects`. |

### Tab 5 — SEO

Identical to Posts: `seoTitle` (optional), `seoDescription` (optional), `ogImage` (optional upload).

### Outside tabs

- `coverPreview` (UI cell) — list-only thumbnail, reuses the Blog `PostCoverCell` pattern (or renames to a generic `CoverCell`).

### Admin config

```ts
admin: {
  useAsTitle: 'title',
  defaultColumns: ['coverPreview', 'title', '_status', 'industry', 'completedYear', 'publishedAt'],
  description: 'Real-world ENVO LED installations. Publish to make a case study visible on the website.',
  group: 'Editorial',   // same group as Blog Posts
}
```

## Hooks

Mirror Posts.ts. Two hooks, no reading-time.

1. **`beforeChange` · autoSlug** — on `create` with empty slug, set `slug = slugify(title)`.
2. **`afterChange` · revalidate ISR** — when public state changes, POST `/api/revalidate?paths=...` with these paths:
   - `/projects`
   - `/projects/<slug>`
   - `/projects/industry/<industry>` for each industry in the project (multi-select → multiple paths)
   - `/projects/<old-slug>` if slug renamed (so stale static page clears)
   - `/projects/industry/<old-industry>` for industries removed in the update (so the old industry's list page refreshes)
   - `/projects/tag/<tag>` for each current tag (and removed tags from `previousDoc`)

Versions/drafts enabled (`versions: { drafts: true }`).

## Routes

Mirror Blog 1:1. Four new/modified routes.

| Route | Action | Behavior |
|---|---|---|
| `/projects` | **Modify** existing | Fetch from Payload. Hero + stats strip + featured strip (conditional) + filter chips + card grid. |
| `/projects/[slug]` | **New** | Single case-study detail page. |
| `/projects/industry/[industry]` | **New** | Filtered list — projects whose `industry` array includes `[industry]`. |
| `/projects/tag/[tag]` | **New** | Filtered list — projects whose `tags` array includes `[tag]`. |

### `/projects` list page sections (in order)

1. **Hero** — title + lede + stats strip (`500+ installations · 60+ countries · 10+ years · 80% repeat`). Stats stay hardcoded copy for now — they're brand claims, not derived from DB count.
2. **Featured** — `featured === true` projects in a large card row. **Conditional render**: if zero featured, the strip does not render.
3. **Filter chips** — `[All] [Retail] [Hotel] [Storefront] [Architectural] [Canopy] [Other]`. Multi-select on client side; clicking a chip filters the grid below.
4. **Card grid** — 3-column responsive grid of `ProjectCard`. Each card: cover · title · excerpt · industry chips · `<location> · <year>` meta.
5. **Empty state** — when DB has 0 published projects: show a friendly "Case studies coming soon — check back as our latest installs land." message instead of an empty grid. (Same on the industry/tag filter routes when their filtered result is empty.)

### `/projects/[slug]` detail page sections

1. **Breadcrumb** — `Home › Projects › <title>`.
2. **Hero** — large cover image; eyebrow shows industry chips; H1 is the title; meta row is `<client> · <location> · <year>`.
3. **Story body** — rich-text rendered via the existing `RichTextRenderer` (reuse from Blog, no duplication).
4. **Gallery** — image grid (2-col on desktop) with caption beneath each image. Skip if `gallery` is empty.
5. **Products used** — list of cards, one per SKU. Each card calls `getProduct(sku)` server-side and shows image + name + datasheet link. SKUs that don't resolve render a muted "ENVO `<SKU>`" pill (no link). Skip the whole section if `productsUsed` is empty.
6. **Testimonial** — large pull-quote block. Skip if empty.
7. **Related projects** — three cards from the same primary industry, excluding the current project. **Conditional render**: skip if fewer than 2 candidates.
8. **CTA** — same "Find your match · Contact" CTA component used elsewhere on the site.

## Implementation order (12 steps)

```
1. Create src/payload/collections/Projects.ts (mirror Posts.ts skeleton + new tab)
2. Register Projects in payload.config.ts collections array
3. Let dev-server hot-reload regenerate payload-types
4. Drizzle Kit schema push — addition-only (new tables), should NOT trigger a destructive y/N prompt
5. Switch homepage <Projects> component (src/components/home/projects.tsx) to Payload fetch
6. Rewrite src/app/(frontend)/projects/page.tsx to Payload fetch (replaces PROJECTS hardcoded data)
7. Create src/app/(frontend)/projects/[slug]/page.tsx
8. Create src/app/(frontend)/projects/industry/[industry]/page.tsx
9. Create src/app/(frontend)/projects/tag/[tag]/page.tsx
10. Create the 4 new components under src/components/projects/
11. Delete src/data/projects.ts (verify no remaining imports first)
12. Manual smoke test: empty state on all 4 routes; create one draft project in admin; verify revalidation fires on publish
```

## Risk register

1. **Drizzle schema-push wedge.** Detached `npm run dev` has no stdin. Adding a new collection is additive (no column drops), so should NOT trigger an interactive y/N. If `/admin` hangs after step 4, follow [`memory/project_admin-wedge-payload-schema-push.md`](../../../.claude/projects/-Users-marketing-Desktop-envo-website-v2/memory/project_admin-wedge-payload-schema-push.md) — confirm intent, then `yes | nohup npm run dev`.
2. **Homepage compile failure during data migration.** `src/data/projects.ts` is imported by both the list page and the homepage `<Projects>` component. Step 11 (delete the file) must come *after* both consumers are migrated (steps 5 + 6). Out-of-order = build break.
3. **Zero-content launch.** Wei may not have any case-study content ready when this ships. Every route needs a graceful empty state. Test all 4 routes against an empty `projects` table before declaring done.
4. **`productsUsed` SKU rot.** Wei may type a SKU that doesn't exist in Akeneo (typo, retired SKU). `getProduct()` returns null — `ProductsUsedList` falls back to a muted "ENVO `<SKU>`" pill so a typo doesn't crash the page.
5. **`industry` multi-select revalidation gap.** If a project's industries change (add Retail, remove Hotel), both new and removed industries' list pages must revalidate. The hook handles this by including both `doc.industry` and `previousDoc.industry` in the revalidate path set.

## Success criteria

- [ ] Wei can create, draft, preview, and publish a Project in `/admin/collections/projects`.
- [ ] `/projects` renders Payload data (not hardcoded), with filter chips and empty state.
- [ ] `/projects/[slug]` renders for any published project, including Gallery and ProductsUsed sections.
- [ ] `/projects/industry/<i>` and `/projects/tag/<t>` static routes work.
- [ ] Publishing a project triggers ISR revalidation for `/projects`, `/projects/<slug>`, and each industry/tag path.
- [ ] All 4 routes render a graceful empty state when no projects are published.
- [ ] No remaining imports of `src/data/projects.ts`; file deleted.
- [ ] `npm run build && npm run start` succeeds with zero new errors.
